# v0.10.0 milestone: Adoption polish and distribution readiness

Scope after v0.9.0:

## Development cycle

- Move source builds, CI builds, and release workflow defaults to `0.10.0`.
- Keep install docs, GitHub Action examples, and package manifests pinned to the latest published release until `v0.10.0` is published.
- Keep `docs/rules.md` regenerated from the active development rule catalog.

## Distribution

- Formalize the Scoop path after the validated local manifest prototype.
- Decide whether HermesScan should use a project-owned Scoop bucket or keep the manifest in this repository first.
- Add CI validation for `packaging/scoop/hermesscan.json` so manifest drift is caught before release.
- Keep the manifest contract test focused on static shape, release URLs, and hashes so CI does not require Scoop.
- Decide whether release automation should open or prepare a manifest update after publishing a new release.
- Treat Homebrew as a follow-up unless distribution becomes the primary release theme.

## Release-integrity rules

- Add a rule for release workflows that publish binaries without `checksums.txt`.
- Add a rule for release workflows that use broader permissions than their release steps require.
- Evaluate a rule for install or download scripts that fetch release binaries without checksum verification.
- Keep release-integrity rules conservative and covered by precision tests.

## Adoption workflow

- Add `hermesscan init --profile minimal|ci|supply-chain` or an equivalent profile mechanism.
- Improve the starter `.hermesscan.json` generated by `hermesscan init`.
- Make baseline and inline-suppression guidance more actionable for teams adopting HermesScan in existing repositories.
- Keep advisory-first adoption examples separate from strict quality-gate examples.

## Rule and report UX

- Evaluate rule metadata extensions for examples or remediation details.
- Improve `hermesscan rules show` output if new metadata makes individual rules easier to act on.
- Consider JSON schemas for `.hermesscan.json` and rule catalogs.
- Add focused tests around config, baseline, and suppression behavior where adoption flows depend on them.
